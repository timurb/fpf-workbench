# Style guide

## Общие конвенции

- Язык публикаций: основной язык для страниц в `docs/**` — русский.
- Английский допускается для точных технических идентификаторов, имен сущностей, путей и общепринятых терминов, когда перевод ухудшает однозначность.
- Для внутренних YAML с человекочитаемыми именами композитов/групп храните оба варианта: `*_name_ru` и `*_name_en`; в human-view публикациях используйте русский вариант.
- Заголовки: H1 один на страницу; дальше — H2/H3.
- Ссылки: только `[Title](relative/path.md)`, без wiki-links и без code blocks для ссылок.
- Структура: каждая рабочая страница должна иметь явные **границы контекста** (что входит/не входит) и **ссылки на источники**.
- Таблицы: если в ячейке перечисляется несколько пунктов, используйте список (`<ul><li>...</li></ul>`), а не перечисление через `;`.

## Обязательное правило: именование в публикациях

- В публикациях (`docs/**`, кроме `docs/work/_internal/**`) используйте человекочитаемые названия вместо `kebab-case`/`snake_case` идентификаторов.
- Если машинные идентификаторы нужны для согласованности или повторяемых вычислений, храните их в `internal`-артефакте (`docs/work/_internal/**`) и ссылайтесь на него из публикации.
- При конфликте между читаемостью публикации и машинной структурой приоритет у читаемости публикации; машинная структура переносится в `internal`.

## Обязательное правило: происхождение чисел

- Во всех страницах `docs/**` каждый числовой показатель (процент, диапазон, порог, коэффициент, FTE, SLA/SLO, срок, лимит) должен иметь явное происхождение.
- Исключения: даты, номера версий, служебные идентификаторы (`GM-FU-01`, `DRR-0060`) и имена грейдов (`L11..L7`), если они не используются как измеряемая метрика.
- Для каждого числа должен быть указан тип происхождения: `carrier-derived`, `public-benchmark` или `working-heuristic`.
- Для `carrier-derived` и `public-benchmark` должна быть ссылка на источник и дата фиксации.
- Для `working-heuristic` обязательно указывать способ получения (формула/правило), исходные допущения и статус калибровки (`uncalibrated`, `partially-calibrated`, `calibrated`).
- Если происхождение не установлено, число помечается как `Unknown` и не используется как основание для `pass`-решений.

Рекомендуемый формат фиксации чисел на рабочих страницах:

| Число/метрика | Значение | Тип происхождения | Источник/метод получения | Статус калибровки |
| --- | --- | --- | --- | --- |

## Шаблон рабочей страницы (рекомендуется)

Используй этот каркас для страниц в `docs/work/**`:

- **Цель**
- **Контекст и границы (scope)**
- **Результат (какой артефакт хотим получить)**
- **Неизвестное / вопросы**
- **Следующие шаги (plan)**
- **Источники / evidence**

## FPF-совместимость (прагматично)

- Различай *carrier* (исходный текст/сообщение) и *description* (наша переработанная формулировка).
- Отдельно фиксируй планы (`WorkPlan`) и фактическое выполнение (`Work`), не используя планы как доказательство выполнения.

См. также: [refs/fpf/FPF-Spec.md](../refs/fpf/FPF-Spec.md).
